# Plan: Router halb deterministisch, Benchmark = Console

Stand: 2026-05-29  
Ziel-Repo: **CaiLama** (Implementierung), **CaiLama-Master** (Aufgabenkatalog/Doku)  
Ausloeser: Router-Rolle scheitert bei ~99 % aller Modelle im Benchmark, obwohl
Produktrollen brauchbar sind. Mistral zeigt bei `small`/`large` gute Qualitaet,
wird aber durch Free-Plan-Limits und den JSON-Router-Vertrag verzerrt bewertet.

## Leitprinzip

**Benchmark und Console testen denselben Produktpfad.**  
Kein paralleler „Prompt-only“-Router mehr. Der Benchmark darf nur messen, was die
Console spaeter wirklich ausfuehrt.

## Ist-Zustand (Luecke)

| Schicht | Console | Benchmark heute |
|---|---|---|
| Eingabe | `InputRouter.route_input()` | nicht genutzt |
| Routing | `ModelRouter.decide()` | nicht genutzt |
| Fallback | `_FallbackRouter` nur bei LLM/Parse-Fehler | nicht genutzt |
| Router-Prompt | `system_prompt.router.md` + Tool-Deskriptoren | direkt `_build_router_prompt()` |
| Ergebnis | `ModelSelection` → QA/Task/Tool-Ausfuehrung | Rohtext → harte JSON-/Tool-Schema-Pruefung |

Der Benchmark prueft damit **LLM-JSON-Formatierung**, nicht den echten
Konsolen-Workflow. Das erklaert die Router-/Task-Massenfehler unabhaengig vom
Modell.

## Soll-Architektur: halb deterministischer Router

```text
Nutzerinput
  -> InputRouter (deterministisch: slash / legacy / leer / frage / aufgabe)
  -> RoutingPipeline (neu, gemeinsam fuer Console + Benchmark)
       1. Deterministische Tool-/Rollen-Erkennung
          (_FallbackRouter._tools_for_query, extract_params, suggest_role_from_query)
       2. Optional LLM-Semantik (nur wenn Schritt 1 unsicher)
          - Intent: role_hint, confidence, reason, disambiguation
          - KEINE verbindliche Tool-Param-Erzeugung durch LLM
       3. Deterministische Plan-Bindung
          - Toolwahl + Argumente aus Query/FEN/PGN/OCR-Pfad/Profil-ID
          - Validierung gegen ToolRegistry
  -> ModelSelection (role, model_name, suggested_tools, suggested_tool_params, routing_source)
  -> bestehende QA/Task/AgentLoop-Pfade
```

### LLM-Aufgabe (neu, eng)

Das LLM liefert nur noch **Semantik**, z. B.:

```json
{
  "role_hint": "vision",
  "confidence": 0.82,
  "reason": "PDF page with diagram; OCR before interpretation",
  "needs_clarification": false
}
```

**Nicht mehr** vom LLM erwartet: exakte `tool_params`, Pfade, Stockfish-Pfade,
FEN-Strings in JSON — das macht der deterministische Binder.

### Deterministische Schicht (primär)

`_FallbackRouter` wird zur **primaeren** Routing-Engine hochgestuft, nicht nur
zum Fehler-Fallback. Regeln bleiben secret-frei und testbar:

- FEN in Query → `get_position_eval` / `evaluate_legal_moves`
- PGN/Pfad/„analyze game“ → `load_pgn` + `analyze_game`
- OCR/PDF/Bild → `analyze_ocr_document` → Rolle `vision`
- DWZ/FIDE → `search_dwz` / `search_fide` → Rolle `researcher`
- Coach/DGT/FEN → `start_coach_session`
- Uebersetzung → Rolle `translator`, keine Tools
- Opening → `get_opening_text`

LLM nur bei:

- widersprüchlichen Signalen,
- niedriger confidence,
- mehrdeutiger Natuerlichsprache ohne harte Marker.

## Benchmark-Anpassung

### 1. Gemeinsamer Entry-Point

Neues Modul, z. B. `cailama.agent.routing_pipeline`:

```python
def route_user_request(
    *,
    settings: Settings,
    query: str,
    parsed: ParsedInput | None = None,
    llm: LLMClient | None = None,
    tools: ToolRegistry | None = None,
) -> RoutingDecision:
    ...
```

- **Console:** `AgentController` ruft nur noch diese Funktion auf.
- **Benchmark:** Router-Probes rufen dieselbe Funktion auf (mit gleichem
  `Settings`, gleicher ToolRegistry, gleichem LLM-Adapter).

Observations exportieren zusaetzlich:

- `routing_source`: `deterministic` | `llm_semantic` | `fallback`
- `deterministic_match`: bool
- `expected_role`, `expected_tools` (nur fuer Benchmark-Vergleich)
- **kein** erzwungenes LLM-JSON als Ground Truth fuer Tool-Parameter

### 2. Router-Benchmark-Katalog neu schneiden

Statt „Modell muss exaktes JSON mit Tool-Args liefern“:

| Pruefung | Bedeutung |
|---|---|
| `expected_role` | Rolle stimmt semantisch |
| `expected_tools` (Namen) | Tool **Klasse** stimmt, Args optional separat |
| `deterministic_tools_match` | deterministische Schicht findet erwartete Tools |
| `llm_disagreement` | LLM widerspricht deterministischem Plan → Warnung, kein Auto-Fail |
| `no_secret_leak` | keine Pfade/Secrets im Export |

Bestehende `router-001` … `router-010` im Master-Katalog bleiben inhaltlich
gueltig (PGN, FEN, OCR, DWZ, FIDE, Translator, Import). Die **Auto-Checks**
wechseln von `expected_tools` (volles Schema) auf:

- `expected_role`
- `expected_tool_names`
- optional `deterministic_plan`

OCR-Beispiel `router-005` wird damit zum echten Projekt-Use-Case:
deterministisch `analyze_ocr_document` + Rolle `vision`, Benchmark prueft
Intent — nicht ob Mistral ein perfektes JSON schreibt.

### 3. Task-Rolle angleichen

Task-Probes sollen den **Task-/Agent-Pfad** der Console nutzen (`PromptBuilder`,
`AgentLoop`, Tool-Validierung), nicht einen separaten Prompt-only-Pfad.
Strukturpruefung bleibt fuer Task-**Output** (Plan/JSON), aber Tool-Ausfuehrung
wird nicht vom LLM-JSON abgeleitet.

### 4. role_contracts bleibt zentral

Bereits vorhanden in CaiLama: `role_contracts.parse_router_response`,
`structure_check`, `catalog_structure_check`.

Aenderung:

- Router-Checks auf **RoutingDecision** statt Rohtext-JSON
- Neue Checks: `role_match`, `tool_name_match`, `deterministic_binding_ok`
- Alte harte `tool_params`-Gleichheit nur noch als **Diagnose**, nicht als
  Benchmark-Pass/Fail fuer Router

## Console-Aenderungen (CaiLama)

1. `ModelRouter.decide()` refactoren → `RoutingPipeline.route()`
2. `_FallbackRouter` → `DeterministicRouter` (primary)
3. Optionaler `SemanticRouterLLM` mit temp=0, kurzem JSON-Intent-Schema
4. `ModelSelection` erweitern um `routing_source`, `confidence`
5. Logging/SessionLogger: Routing-Entscheidung strukturiert speichern
6. `system_prompt.router.md` anpassen: Semantik statt Tool-Param-JSON

## Mistral (Pro-Plan)

Operativ:

- Router-Backend `mistral-api` bleibt bei `max_concurrent_requests: 1`
- Nach Plan-Upgrade Cooldown abwarten bzw. Router neu starten
- Erwartung: keine 6-Minuten-Retry-Spikes mehr, Median weiter ~1–5 s

Rollen-Fit laut Zwischenlauf:

| Rolle | Einschaetzung |
|---|---|
| `small` | 10/10 — starker Kandidat |
| `large` | 10/10 — starker Kandidat |
| `router` | aktuell verzerrt durch JSON-Benchmark, semantisch testen |
| `vision`/OCR-Routing | vielversprechend (`router-005`, OCR-Pipeline) |
| `translator` | noch offen im Lauf, sprachnah |

Mistral gehoert in die **OCR-/Vision-/Translator-Spur**, nicht als Router-JSON-
Champion.

## OCR (Projektbezug)

Bestehende Linie beibehalten und im neuen Router explizit verankern:

- Tool: `analyze_ocr_document`
- Rolle: `vision`
- Benchmark: Text-OCR-Unsicherheit + **Routing-Intent** „PDF/Diagramm“
- Live-OCR-Benchmark (6 PDFs, FEN-Gates) bleibt separater Qualitaetsnachweis

Naechster Schritt nach Router-Align:

- Vision-Benchmark optional um **echte OCR-Artefakte** erweitern (ohne Secrets),
  nicht nur Text-Fall „unsichere OCR-Zusammenfassung“.

## Text-to-Speech (Zukunft, Coach-Feature)

Kein Bestandteil dieses Router-Refactors. Als **downstream Output-Kanal**
planen:

| Aspekt | Vorschlag |
|---|---|
| Trigger | `/speak`, Coach-Antwort, Training-Card-Vorlesen |
| Rolle | nicht im LLM-Router; post-processing auf `coach`/`scribe`-Output |
| Pipeline | `text_for_speech` (bereinigt, ohne FEN/JSON) → TTS-Backend |
| Benchmark | eigene kleine Rolle oder Coach-Subcheck: „speech-safe summary vorhanden“ |
| Backends | lokal (piper/espeak) oder API; Router-unabhaengig |

TTS und Router entkoppeln: Routing entscheidet **was** gesagt wird, TTS **wie**
es ausgegeben wird.

## Akzeptanzkriterien

1. Dieselbe Query liefert in Console und Benchmark dieselbe Routing-Entscheidung
   (modulo explizit dokumentierter LLM-Unsicherheit).
2. Router-Benchmark misst **Intent**, nicht JSON-Schreibfaehigkeit.
3. OCR-Route (`analyze_ocr_document` → `vision`) ist deterministisch und
   benchmarkbar.
4. Bestehende Produktrollen (`small`, `large`, `coach`, …) behalten ihren
   PromptBuilder-Pfad unveraendert.
5. Mistral-Pro kann den laufenden Rollenbenchmark ohne Cooldown-Artefakte
   abschliessen.

## Offene Entscheidungen (Torsten)

1. LLM-Semantik komplett abschalten, wenn deterministisch `confidence >= X`?
2. Sollen Router-Probes kuenftig **ohne** LLM-Call laufen duerfen, wenn
   deterministisch eindeutig (schnellerer Benchmark)?
3. TTS zuerst lokal (offline) oder API (z. B. Mistral/ElevenLabs)?
